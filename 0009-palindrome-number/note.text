是的，sliding window 非常適合這個場景。讓我解釋為什麼，以及它怎麼運作。

為什麼適合？
你的需求是「任意 10 秒區間內，登入次數超過閾值就 block」。這正是 sliding window 要解決的問題——它不會像 fixed window 那樣在邊界處產生漏洞。

Fixed Window 的問題
假設閾值是 5 次 / 10 秒，用 fixed window（例如每 10 秒一個桶）：

第 1 個窗口的第 8~10 秒：打了 4 次（沒超標）
第 2 個窗口的第 0~2 秒：又打了 4 次（也沒超標）
結果：在真實的 4 秒內打了 8 次，卻完全沒被擋住。

Sliding Window 怎麼做
有兩種常見做法：

1. Sliding Window Log（精確）

把每次登入的 timestamp 都記下來。每次新請求進來時：

丟掉所有超過 10 秒的舊紀錄
數剩下的紀錄數量
如果 ≥ 閾值 → block
優點是完全精確，缺點是要存每一筆 timestamp，記憶體用量跟請求量成正比。不過因為你的場景是「登入」而不是一般 API 呼叫，頻率本來就不高，所以這個做法很實用。

2. Sliding Window Counter（近似）

結合兩個相鄰的 fixed window 做加權估算：

假設當前窗口已過了 30%
估算值 = 當前窗口計數 + 上一個窗口計數 × 70%
優點是只需存兩個計數器，非常省記憶體，缺點是結果是近似值。

登入場景的建議
對登入防暴力破解來說，Sliding Window Log 最合適，原因是：

登入頻率遠低於一般 API，存 timestamp 的成本很低
安全場景需要精確判斷，不能有漏網之魚
實作上用 Redis 的 sorted set 就能搞定（timestamp 當 score）

class HitCounter:
    def __init__(self):
        self.hits = []
        self.l = 0
    def hit(self, timestamp: int) -> None:
        self.hits.append(timestamp)

    def getHits(self, timestamp: int) -> int:
        while self.l < len(self.hits) and timestamp - self.hits[self.l] >= 300:
            self.l += 1
        return len(self.hits) - self.l

對，因為 deque 底層不是一整塊連續記憶體。

List 的情況： 底層是一個連續的 array，list[i] 直接算 起始位址 + i × 元素大小 就跳過去了，所以 O(1)。

Deque 的情況： CPython 的 deque 底層是一串固定大小的 block（每個 block 存 64 個元素），block 之間用雙向鏈結串起來。要存取第 n 個元素，得先算出在哪個 block，再走過去，所以是 O(N/64)，還是 O(N)。

簡單想就是：

list:    [0][1][2][3][4][5]...  ← 連續記憶體，算位址直接跳
deque:   [block0] ↔ [block1] ↔ [block2]...  ← 要一格一格走過去
所以如果你的場景需要大量 deque[i] 隨機存取，那用 deque 反而比 list 慢。deque 的優勢就只在兩端操作。

class HitCounter:
    def __init__(self):
        self.hits = [0] * 300
        self.counts = [0] * 300
    def hit(self, timestamp: int) -> None:
        index = timestamp % 300
        if self.hits[index] != timestamp:
            self.hits[index] = timestamp
            self.counts[index] = 1
        else:
            self.hits[index] = timestamp
            self.counts[index] += 1


    def getHits(self, timestamp: int) -> int:
        total = 0
        for i in range(len(self.hits)):
            if timestamp - self.hits[i] < 300:
                total += self.counts[i]
        return total

推導卡特蘭數與時間複雜度的數學關係。
這題的時間複雜度推導分兩步：

第一步：有多少個合法組合？

n 對括號的合法組合數就是第 n 個 Catalan number：

C(n) = C(2n, n) / (n + 1)
例如 n=3 → C(3) = C(6,3)/4 = 20/4 = 5 組。

第二步：Catalan number 有多大？

用 Stirling's approximation 可以估出：

C(n) ≈ 4^n / (n^(3/2) · √π)
省略常數 √π，就是 O(4^n / n^(3/2))。

第三步：乘上每組的成本

每組結果長度是 2n，"".join(stack) 要花 O(n) 把它組成字串，所以總時間：

O(n × C(n))
= O(n × 4^n / n^(3/2))
= O(4^n / n^(1/2))
= O(4^n / √n)
所以最終結果是 O(4^n / √n)。

面試時不需要推完整個過程，只要能說出「合法組合數是 Catalan number，大約是 4^n / n√n，每組花 O(n) 建構，所以總共 O(4^n / √n)」就夠了。

2. 不能轉字串 → 需要用數學方式同時取最高位和最低位來比較
3. 先用 divisor 找到最高位的權重（例如 12321 → div = 10000）
4. 每輪比較頭尾，相同就去頭去尾，縮小數字繼續比
5. 全部比完都相同 → 是回文

數學操作速查表（這題的核心）：
  取尾巴  →  x % 10          12321 % 10     = 1
  取頭    →  x // div        12321 // 10000 = 1
  去頭    →  x % div         12321 % 10000  = 2321
  去尾巴  →  x // 10         2321 // 10     = 232
  去頭去尾 → (x % div) // 10  合併上面兩步
  div 縮兩位 → div /= 100     因為頭尾各去掉一位

踩過的坑：
1. divisor 的建立方式不熟。記法：從 1 開始，只要 x > div * 10 就繼續乘 10，跳出迴圈時 div 就是跟 x 同位數的最大 10 的冪。
2. 去頭去尾的數學操作組合不直覺，需要熟記上面的速查表。關鍵是理解 % 是「去高位保低位」，// 是「去低位保高位」，兩者互為反向操作。

複雜度（n = 位數）：
時間 | O(n)，每輪去掉兩位數，共跑 n/2 輪，省略常數為 O(n)
空間 | O(1)，只用幾個變數
